Design Review For PowerManager

 

Review DateOct 21, 2016

Team

Mike Austin, CoreTech
Feature 
StateDRAFT
Technical OwnersMike Austin

 

Product Owners

Varun Chopra (indirect owner)

Reviewers

Alex Koff, Masato Ikura, Darren Tsung ...

QA Contact 


I- Problem to be Solved 

Summary

Our games don't need to be run at 60 FPS all the time.  It wastes the player's battery and makes their device warm or even hot.  In the extreme, it can also throttle performance.

Depending on the device and game / application, battery usage can vary.  With PowerManager, we can also also monitor the gains in using reducing power.

We want to be able to choose frame rate dynamically through hints in code, or statically by user preference or battery level.


Goals

Provide a class that allows developers to set power usage modes, and give hints through code when performance or power saving is wanted.

It should provide the ability to add incremental power improvements: Course grained at first (e.g. Core loop), the fine grained (idle core loop).

It should also provide options such as user-selectable power mode, and dropping to power saving if the battery is below 20%, for example.

The implementation time with examples in one game with a few hints should be short – under 1 week.  Other games can follow.

II- Requirements Covered by this Design


Features

PowerManager with modes:

  • Performance – run the game at highest frame rate (sent from server or set in client)
  • PowerSaving – run the device at lower frame rate
  • Dynamic – run the device at a lower frame rate, but switch to performance temporarily based on code hints

With dynamic mode, hints can be given through code:

  • void PushDynamicPerformanceHint() – Run in performance mode if possible until changed (Core game loop, etc.)   
  • void PushDynamicPerformanceHint(float seconds) – Run in performance mode for a few seconds (Dialog animations, panning / zooming)
  • void PushDynamicPowerSavingHint() – Run the game in power saving mode (Level pick dialog, Settings dialog, End Game dialog)
  • void PopDynamicHint()

Dynamic mode can default to power saving if the player's battery is low.


Details

I started prototyping with Detective Match.  Hinting on touch, map scroll, dialog open and close.

The ability to switch between modes would be available to all games.  Since most game logic is in game specific classes, adding hints for dynamic mode would be need to be done per game.

User Stories

As a player, I want my battery to not be drained quickly when playing casual games.  Also, I don't want my device to get hot and loose performance because of throttling.

As a developer or qa tester, I would like the ability to test battery usage in my games.  I want to be able to switch between 30, 60, and dynamic FPS in game.

VIII- Rollout / future adoption plan

  1. How can this feature be enabled / disabled?
    Setting PowerMode to Performance (server or client configuration) uses the original/default implementation.

  2. If this is a client version gated feature, how do we handle multi-device login where players can login with old client version again after they installed new version?
  3. How is it configured?
  4. How will it go live?
  5. What training needs to happen before the feature goes live?
  6. Will there be more phases to deploying this feature with future requirements?
  7. How will development be isolated and not interact with other features?
  8. What parameters can we experiment on?
  9. Will engineers configure the feature, and if not are there tools required and is the configuration / feature user-friendly enough?
  10. If non-engineers will configure, what is the handoff process to their team?
