Code Review Checklist

Skip to end of metadata
  • Created by Mike Austin, last modified on Dec 04, 2015
Go to start of metadata

This is a high-level checklist to be used when doing code reviews. Add items as you see fit in the form of questions. Please talk to Mike Austin or Alex Koff regarding major changes to this page.

 

In general:

  • Check-ins should be small, incremental, and dormant if it's part of a larger feature.
  • Design reviews should be done periodically, before code reviews and commits.
  • We support iPhone 3GS, iOS 5.1.1 and greater, so occasional device testing is a must.

Pre-Check

Thing to check for before starting the code review:

  • ✅ Is it checked for multitasking in and out, device rotation, and low memory situations?
  • ✅ Does it work on iPhone 3.5", 4.0" and iPad devices? What about iPhone 6+ and iOS 8?
  • ✅ If the review is for base code, does it affect other games - compilation errors or crashes?
  • ✅ Was the Xcode static analyzer run to check for issues beyond basic compilation errors?
  • ✅ Are notes or documents available for QA if they are testing a specific feature?

General

Overall things to check for:

  • ✅ Is code clear and readable, with minimal cleverness and use of the ? conditional operator?
  • ✅ Does code have loose coupling, private properties, use internal logic, and prefer interfaces?
  • ✅ Does code have small, cohesive classes, short methods, few arguments, with good abstraction?
  • ✅ Is logic code in controllers that the UI can query - for example, canAdvanceToNextLevel?
  • ✅ Is shared code modular and abstracted, and have the ability to be composed and unit tested?
  • ✅ Does code avoid using introspection such as isKindOfClass and NSClassFromString()?
  • ✅ Does code avoid using language specific features such as addObserver:forKeyPath:?

Coding Conventions

Things related to code conventions and style:

  • ✅ Does code use well named methods and variables, use Objective-C literals and use instancetype?
  • ✅ Do overridden methods call super in most cases? When they don't, is it a composition of another?
  • ✅ Does code use standard, consistent syntax, double for times, 'f' for floats, '.' for properties, etc.?
  • ✅ Do classes have brief descriptions, pragma'd groups of methods, optionally have simple examples?
  • ✅ Does code prefer ConfigManager, AppConstants and static const over #define and #if statements?
  • ✅ Does code prefer NSInteger/NSUInterger over int/uint?

https://wiki.storm8.com/index.php/ObjC_Style_Guide

Memory Management

Things related to memory management and resource tracking:

  • ✅ Does it use retain/assign properties and autorelease properly and handle lowMemoryWarning?
  • ✅ Does it clean up NSTimers, GameLoopTimers, notifications and injected views in viewWillDisappear/didJumpOutOfPage?
  • ✅ Is transient data released, are added views removed, and tables unloaded in didJumpOutOfPage?
  • ✅ In Unity, is memory freed explicitly in tight loops and at pauses - loading game screen, etc.?

https://developer.apple.com/library/ios/documentation/Cocoa/Conceptual/MemoryMgmt/Articles/MemoryMgmt.html

http://docs.unity3d.com/Manual/UnderstandingAutomaticMemoryManagement.html

Low End Devices

Things related to low end devices with small memory:

  • ✅ Does it use isHighEndDevice, nonCachedPngImageNamed:, nils out properties between use?
  • ✅ Does it use viewDidAppear to delay heavy work until the dialog is visible to reduce stutter?
  • ✅ Does it use 9-Slicing and composition to reduce binary and image memory usage?

Storm8 Shared Code

Things related to code re-use and available libraries:

  • ✅ Does code use S8ScalingView, Animator, S8Action, S8Label, S8Button, VerticalStackView, etc?
  • ✅ Does code use StringUtil, DictionaryUtil, ViewUtil extensions, etc. over re-inventing the wheel?
  • ✅ Does the change refer to any game-specific code that will break when running on other projects?
  • ✅ Does the change contain new files that should be included into other projects with the xcode project script?

http://peeps.storm8.com/docs

Thread Safety

Things related to thread safety:

  • ✅ Does code use the dispatch_once() thread-safe pattern in singleton initialization and related code?
  • ✅ Are non-thread-safe blocks wrapped in a call to dispatch_async(dispatch_get_main_queue(), ...)?

iOS Compatibility

Things related to iOS version compatibility:

  • ✅ Are iOS > 5.1.1 method calls wrapped in a respondsToSelector?
  • ✅ Are optional protocol method calls wrapped in a respondsToSelector?

Server API

Things related to talking to the server:

  • ✅ Do all ChangeEvents have unique abbreviations? Are API calls done in Controllers and Managers only?

Server Code

Things related to back end server code:

  • ✅ For data-loading objects: are they inheriting from the most appropriate class (DbObjectBase vs McObjectBase, McDerivedObjectBase vs McDerivedLocalObjectBase, Cached* versions, etc...)
  • ✅ Check for usage of empty() vs isset(): empty is more general, and catches empty arrays, empty strings, 0 values, etc... in addition to just null
  • ✅ Check for correct usage of === vs ==
  • ✅ Does the code need to be version-guarded to not break older clients (check this especially when changing format of data returned to clients)
