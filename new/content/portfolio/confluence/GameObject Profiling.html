GameObject Profiling

Skip to end of metadata
  • Created by Mike Austin, last modified on Oct 07, 2016
Go to start of metadata

Summary

The number of GameObjects in a scene affects performance in many ways.  It's not always a 1:1 correlation because of batching, object culling, etc. but in general the fewer the better.  The number of object may affect:

  • Average frames per second
  • Speed of GameObject.Find()
  • Memory usage and fragmentation

 

To reduce the number of GameObjects in a scene, there are several alternatives:

  • Use the Flyweight design pattern / god object.  Create a MonoBehavior that delegates tasks to a collection of other objects or methods.
  • Use CoroutineHelper if you need to start a coroutine from a non-MonoBehavior class.
  • Use meshes instead of individual child GameObjects if they use the same texture.
  • Use ScriptableObject if you need the ability to edit data visually.

Debugging Tool

You can add the DebugObjectLogger component from s8unity_shared to any object in a game scene to monitor the number and types of GameObjects in the scene.  Setting MaximumDepth allows you to choose the object hierarchy depth you want printed.  After the max depth has been reached, no child objects are printed in the log, but the number of descendants (full depth) is shown.

You can add other interesting objects to count as you see fit.  Also, the convenience methods are static so can be used outside of the class (currently using them in performance tracking).
