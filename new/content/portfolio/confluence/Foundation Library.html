Design Review For Client Foundation Library

 

Review Date Oct 27, 2016

Team

CoreTech
FeatureClient Foundation Library
StateREVIEW
Technical Owners
Mike Austin

 

Product OwnersMike Austin
ReviewersAlex Koff, Jack Cheung, Gordon Szebenyi
QA Contact 


I- Problem to be Solved 

 

Description

Microservices should have very few dependencies.  On the other hand, code needs to be shared to benefit from new / enhanced features, bug fixes, and generally avoid fragmentation.

Foundation is a collection of very core classes: extensions, utils, and other classes that have very broad use, and don't have any external dependencies.


Goals

The goal is to be able to use shared code, but still have control over when and what changes are included.  It should be easy to add to a game or microservice, and easy to update.

Because code is not just copy-pasted, it provides better structure for updating, contributing, and ensuring units tests are added.  It also allows documentation in one central place.

COR-1549 - Finalize Core.Foundation sharing procedure and host a few first classes Closed

II- Requirements Covered by this Design

The main focus of this design is how to deploy the shared code.  What goes in will likely be driven by clients that use it.

The first client will be Realtime Chat, as it's the first client microservice.  Games can also use it going forward, but none have been targeted at this point.

Setup requires only an initial git command, and can be updated or pushed to feature branch for a pull request with one more git command.

 

User stories:

As a lead or staff engineer, I want to use shared code in our project but also have control over when it's updated. It should be simple to check out and update when there are new features or bug fixes.  It should also be simple to contribute new code.  There should be no external dependencies in the shared code.

As an engineer on a team, I want to used shared code, but not have to sync or update a separate repo manually.  Everyone else on the team should be using the exact same revision of the shared code.

III- Known Future Requirements

In the future, we need to define what belongs in the library.  Gray areas arise with AssetManager and other larger "subsystems" vs just utility and independent classes.  There is a potential for creating a new AssetManager that doesn't have external dependencies (using interfaces), and is more easily unit tested that could be a part of Foundation.  That requires much more work than simply copying over classes such as DeviceUtils.

IV- Use Case Scenarios

A team working on a new microservice wants shared code, but doesn't want to be affected by automatic updates.  They also don't want to copy & paste code into the project and cause fragmentation.

It's simple to add Foundation using Git subtree:

    $ cd /Users/mike/Projects/storm8/game
    $ git subtree add --prefix game/Assets/Foundation http://mike@stash.storm8.com:7990/scm/tool/s8unity_foundation.git master

The files are now part of the project, and everyone is using that revision of the pull.  A staff or lead engineer would pull from the repo when they want the latest version (or even a specific revision):

    $ git subtree pull --prefix game/Assets/Foundation http://mike@stash.storm8.com:7990/scm/tool/s8unity_foundation.git master

Since the files are copied to the project, enhancements can be made to Foundation local to the project.  Changes can easily be pushed back the the Foundation repo with a pull request:

    $ git subtree push --prefix game/Assets/Foundation http://mike@stash.storm8.com:7990/scm/tool/s8unity_foundation.git NewFeatureName
    $ open http://stash.storm8.com:7990/projects/TOOL/repos/s8unity_foundation/pull-requests

You can also add the subtree as a remote to shorten the command, but it's not needed.  How to update the repo is described in the _README.txt in the root of the repo.

V- Solutions Considered and Tradeoffs Involved

Experimented with using DLLs, manual merging, and Git subtree.  One question was how to version the library – how to make changes without possibly breaking clients.  The conclusion is that Foundation is a set of miscellaneous classes that don't fit well into a versioning scheme.  For example, adding a method to a class – is that now version 1.1?  Using Git subtree allows ad-hoc updates, giving more freedom to the game or microservice.

Versioning also comes up when a microservice and a game that uses that microservice wants two different versions of a library. In the end, it's much simpler if the same Foundation is used for both.  This requires that classes not change often or at all, other than additional methods.  It also happens to be a good rule for Foundation in general.  Other options are to use namespace versioning such as Foundation.V1_1, but as noted above, versioning miscellaneous classes becomes difficult and requires more coordination.


Looking at the 3 ways to share code for Foundation (some good for different purposes):

  • Repo (best for s8unity_shared)
    • Always have the latest code after a pull.  We don't necessarily want the latest code.
    • Not possible to version since each developer has the repo checked out separately.
  • DLL (best for microservices)
    • DLLs don't allow for method inlining and other optimizations.  Not suited for small, repetitive methods.
    • DLLs are packaged and well suited for versioning.  Foundation is a collection of random classes with no milestones.
    • DLLs make it difficult to contribute back to the source.  Requires cloning source repo and creating a symlink.
  • Git subtree (best for foundation)
    • Leads, staff need to learn git subtree add/pull.  Commands are simple, should not be an issue.
    •  Code lives in game, no need for each developer to check out separate repo.
    • Allows ad-hoc versioning, pulling the latest changes or revision from the repo.
    • Allows extending foundation locally, with the option to push upstream with a pull request.

VI- Chosen Solution Description

  1. Overview 
    Git subtree works well for Foundation because of the ability to update only when you want, storing copies of files locally for additions and optimizations, and less maintainance and coordination.
     

  2. Requirements

    1) All classes must be unit tested. This is core code that may be shared with all games and all microservices.

    2) All classes should be well documented. Every class should have a comment as well as any non-trivial methods.

    3) External dependencies are not allowed. Interfaces defined in Foundation should always be used for dependencies.

    4) All changes need a pull request and approval from a lead engineer, staff engineer, or architect.

  3. Describe how this is the simplest approach allowing for future extensibility
    Out of the 3 options for sharing code for very core classes that don't change often or at all, Git subtree has proven to be the easiest solution.  It only requires the lead or staff engineer to update when wanted.  Other engineers just use the library transparently.  If a team wants to add code to Foundation, it's easy to develop locally, then push to a FeatureBranch for a pull request.  Foundation as a DLL has many issues: it's not well suited for versioning / milestones, and it makes it difficult to push changes upstream.  Foundation as a symlink requires everyone on the team sync to the same revision of the repo.  It's more work, and more error prone.
     
  4. Tools required
    Git subtree works with Sourcetree or git command line
     
  5. Git subtree details
    "Subtrees are not to be confused with submodules, which are meant for the same task. Unlike submodules, subtrees do not need any special constructions (like .gitmodule files or gitlinks) be present in your repository, and do not force end-users of your repository to do anything special or to understand how subtrees work. A subtree is just a subdirectory that can be committed to, branched, and merged along with your project in any way you want."

    https://github.com/git/git/blob/master/contrib/subtree/git-subtree.txt

VII- Testing Approach

  1. Automated Testing
    All classes in Foundation should have unit tests.  Testing Foundation itself will be done by using Realtime Chat as the first client.
     
  2. QA Testing Coverage
    Testing is driven by unit tests, since the feature is simply shared code.  We should monitor any games that switch to foundation, but not expect any issues.

VIII- Rollout / future adoption plan

  1. Realtime Chat will be the first adopter, as well as any other client microservices.  It will be introduced at tech sync, and I'll mention in an email to eleads.  It will be mentioned in bi-weekly architect emails to e.
  2. What training needs to happen before the feature goes live?
    1. Some light training on how Git subtree works: How to add, pull and push changes.  Also talking about the requirements of unit testing and documentation.

X- Review Action Items

  • Make sure leads / staff review git history of Foundation to see if they want to pull changes.  Weekly?
  • Make sure teams treat local branch of subtree like repo – same rules as Foundation repo
  • All commit to local branch also require pull requests?  If so, remove Foundation repo pull request requirement?
  • Can we set permissions on local branch subtree so people don't add code without some thought?
  • Should we require each class be in it's own namespace, to show explicit inter-dependencies?
  • Move Tools/Foundation to Unity/Foundation and add a few more classes such as S8Event
  • Who should be responsible for adding unit tests to S8Event?  Main authors, or people who need it?
